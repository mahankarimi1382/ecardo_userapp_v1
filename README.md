# Audit Report: ecardo_userapp_v1 (User App)

## Executive Summary

The repository is a mature, fully-built Flutter fintech "user app" originally branded **Qunzo** and currently being re-skinned to **ecardo** (domain `ecardo.ir`). The codebase is large (~448 Dart files, 21 feature modules, ~1,700 localization accessors, 280 bundled assets, full multi-wallet/transfer/P2P/virtual-card/payment-links flows) and follows a consistent GetX (state + DI + routing) + Dio + flutter_screenutil architecture. However, the rebrand to "ecardo" is **only ~5% complete** (package name, Firebase project, app label, Dart class name, manifest label and 3 ARB strings still say "qunzo"/"Qunzo"), **localization only ships English + Arabic** while the target markets need fa/zh/tr/ru as well, **no region-specific fonts** (Persian/Cyrillic/CJK) are bundled, the app uses **only Gregorian dates** with no Jalali/Hijri support, and there are several **serious security issues**: release builds are signed with the debug keystore, the user's plaintext password is persisted in `SharedPreferences` for biometric re-login, the bearer token lives in `SharedPreferences` (not `flutter_secure_storage`), there is no certificate pinning, no ProGuard/R8 rules, and no `network_security_config`. Push notifications use FCM only — which **does not work in mainland China** and is unreliable in Iran. None of the required local payment gateways (Iranian, WeChat/Alipay, YooMoney/Mir, iyzico/PayTR, PayTabs) are present; only Stripe and BsiCards are wired up.

## 3. Fonts & Typography

**Bundled fonts** (`assets/fonts/`):
- `PlusJakartaSans-VariableFont_wght.ttf` (174 KB)
- `PlusJakartaSans-Bold.ttf` (95 KB)
- `PlusJakartaSans-ExtraBold.ttf` (95 KB)
- `PlusJakartaSans-Medium.ttf` (95 KB)
- `PlusJakartaSans-SemiBold.ttf` (95 KB)

**`fontFamily` set in `lib/src/app/config/theme/light_theme.dart` line 8:**
```dart
fontFamily: "Plus Jakarta Sans",
```
Registered in `pubspec.yaml` lines 91–102 as the `Plus Jakarta Sans` family with weight variants 500/600/700/900.

**Font character-coverage analysis** (verified with `fc-query`):
- Plus Jakarta Sans supports **Latin only** (charset rows 0–3 cover Basic Latin, Latin-1 Supplement, Latin Extended A/B; lang list includes `tr` for Turkish — which uses Latin + ı/İ — but **NOT** `ru`, `ar`, `fa`, `zh`).
- Critical gaps:
  - **Persian (fa) — unsupported** (Arabic script, extended Persian letters پ چ ژ گ ک ی)
  - **Arabic (ar) — unsupported** (already in app but relying on system fallback fonts which look terrible / inconsistent across devices)
  - **Chinese (zh) — unsupported** (CJK ideographs)
  - **Russian (ru) — unsupported** (Cyrillic block)
  - **Turkish (tr) — supported** (Latin Extended has ı, İ, ş, ğ)

**Required additions** (must be bundled, not relied on as system fallbacks, for consistent rendering):

| Script | Recommended font | License |
|---|---|---|
| Persian / Arabic | **Vazirmatn** (modern, complete fa+ar coverage) or **Noto Sans Arabic** | OFL |
| Chinese (Simplified) | **Noto Sans SC** (subset by frequency) or **HarmonyOS Sans SC** | OFL |
| Russian (Cyrillic) | **Inter** (Latin+Cyrillic) or **Noto Sans** Cyrillic subset | OFL |
| Turkish | Plus Jakarta Sans already works | — |

Recommended approach: register a multi-family `fontFamilyFallback` per locale in the theme, or use the `ThemeData` `fontFamily` + `textTheme` overrides keyed to the current locale. Plus Jakarta Sans's variable font is only 174 KB so it should be kept; the new fonts will add ~1.5–3 MB (CJK is the big one — recommend subsetting).

**Sample text styles in widgets** show no per-locale font fallback: every widget relies on the global `fontFamily` set in `LightTheme`. There is no `TextStyle.fontFamily` override anywhere for non-Latin locales — meaning Arabic/Persian/Chinese/Russian text will currently render with whatever the OS picks (on Android: Noto Naskh Arabic, Roboto for Cyrillic, Noto Sans CJK for zh). The result is jarring visual mismatch.

    brightness: Brightness.light,

This is a **maintainability and theming-consistency problem**: there is no design system token, every widget hard-codes dimensions.

- `values-night/styles.xml` and `values-night/` Android resources exist (so the **native Android launch theme respects system dark mode**) but the Flutter UI does not.

**No `dark_theme.dart`, no `theme_controller.dart`, no `Get.isDarkMode` usage anywhere.** The end-drawer has a `isSwitchMode` RxBool in `home_controller.dart` line 40 that is **declared but never used** — a leftover stub.

**Color palette (`lib/src/app/constants/app_colors.dart`):** extremely minimal — 11 colors:
- `lightBackground` `#F8F8F8`, `lightPrimary` `#7445FF` (purple), `lightTextPrimary` `#2D2D2D`, `lightTextTertiary` (60% alpha)
- `error` `#DC3C22`, `warning` `#FFAA00`, `success` `#14AE6F`, `white`, `black`, `transparent`

**Recommendation:** expand `AppColors` to a full Material 3 `ColorScheme`, build a real `ThemeData` with `textTheme` and component themes, and add a dark theme + `ThemeController` persisted to `SharedPreferences`.

## 5. Network Layer

**`lib/src/network/service/network_service.dart`** — Dio-based, 526 lines.

**Two Dio instances:**
- `_dio` — used for authenticated calls; has interceptors.
- `_globalDio` — used for unauthenticated calls (login, register, settings, languages, countries); **no interceptors** (just default headers).

**Base URL:** hard-coded in `lib/src/network/api/api_path.dart` line 3:
```dart
static const String baseUrl = 'https://ecardo.ir/api';
```
- **No dev / staging / prod environments.** No `--flavor`, no `String.fromEnvironment`, no `.env` file. Single hard-coded production URL.

**Interceptors** (only one, on `_dio`):
- `onRequest`: injects `Authorization: Bearer <accessToken>` from `TokenService`.
- `onError`: logs 401 ("Unauthorized. Please login again.") but **does nothing else** — no token refresh, no retry, no redirect to sign-in (the redirect happens only inside `_handleDioErrorResponse` which renders an in-app dialog and routes to `signIn`).
- **No retry interceptor** (no exponential backoff).
- **No cache interceptor** (no `dio_cache_interceptor`, no ETag handling).
- **No logging interceptor** (only ad-hoc `debugPrint` calls gated on `kDebugMode`).
- **No certificate pinning** (no `dio_certificate_pinning`, no custom `HttpClient` with `BadCertificateCallback` logic — i.e., default Dio trust behavior).

**Timeouts:** **NONE configured**. `_dio.options.connectTimeout / receiveTimeout / sendTimeout` are all left at Dio defaults (which are effectively infinite for connect, infinite for receive). The only timeout handling is in `_handleDioException` which catches `DioExceptionType.connectionTimeout` and `receiveTimeout` *after they fire from the OS-level socket timeout* — meaning users can wait 30–60+ seconds on a hung connection. **Recommend setting explicit `connectTimeout: 15s, receiveTimeout: 20s, sendTimeout: 20s`.**

**Token storage (`lib/src/network/service/token_service.dart`):**
```dart
SharedPreferences prefs = await SharedPreferences.getInstance();
accessToken.value = prefs.getString(accessTokenKey);
```
- **INSECURE.** SharedPreferences on Android is an XML file in `/data/data/com.qunzo.user/shared_prefs/` — readable on rooted devices, exposed by backup-extraction tools, and **not encrypted at rest**. Should be `flutter_secure_storage` (Keystore-backed on Android, Keychain on iOS).

**Other findings:**
- Every API method (`get`, `post`, `put`, `delete`, `globalGet`, `globalPost`) is wrapped in identical try/catch boilerplate calling `_handleDioException` — heavy duplication.
- `_setupInterceptors()` is **called twice** (line 46 from `_configureHttpClient`, then again at the end of `login()` and `register()` after `_dio.interceptors.clear()` on line 89). The interceptor-clear-then-readd dance in `login()` is suspicious — it temporarily removes auth from all in-flight requests during login.
- HTTP status 401 in `_handleDioErrorResponse` (lines 401–477) renders a custom dialog with `Get.dialog()` and routes to `signIn` — this works but means **any 401 on a background API call will hijack the UI** even if the user is mid-typing in a form.
- Response parsing: `response.data as Map<String, dynamic>` will throw on non-JSON responses (e.g. HTML error pages from a CDN) — no content-type check.
- Status 503 routes to `maintenanceMode` (good).
- Status codes `301/302/304/401/403/404/422/500/503` are handled; everything else falls to default with generic toast.

**Endpoint inventory (`api_path.dart`, 214 lines):** 80+ endpoints covering auth, dashboard, wallets, transactions, QR, add-money, payment, request-money, cash-out, withdraw, transfer, gifts, exchange, profile, password, KYC, referral, 2FA, passcode, notifications, support tickets, beneficiaries, bill payments, virtual cards (BsiCards + Stripe), payment links, gift cards, and P2P trading. All paths are consistent RESTful patterns under `/user/...`.

## 6. State Management & Architecture

- **GetX 4.7.2** — used for **state management** (`Rx`, `Obx`, `.obs`), **dependency injection** (`Get.put`, `Get.find`, `Bindings`), and **routing** (`GetMaterialApp`, `Get.toNamed`, `Get.offAllNamed`).
- **Pattern per feature**: `presentation/screens/<feature>/{controller,model,view}` triad, with `view/` further split into `sub_sections/` for composite screens.
- **Bindings** (`lib/src/app/bindings/app_bindings.dart`, 619 lines) register all controllers per-route; `routes_handler.dart` (422 lines) maps 80+ routes to `(page, binding)` pairs.

**Sample screen — `home_screen.dart`** (122 lines): StatefulWidget with `Get.find<HomeController>()`, uses `Obx` to react to `isLoading`, calls `homeController.loadData()` (no `onInit`-triggered fetch in the controller — wait, `HomeController.onInit` *does* call `loadData()`), shows `HomeSkeletonLoader` while loading, then `RefreshIndicator` + `SingleChildScrollView` with sections (`TopHeaderSection`, `MyWalletSection`, `OtherServicesSection`, `RecentTransactionsSection`). Clean and idiomatic.

**Sample controller — `home_controller.dart`** (348 lines): typical GetX pattern — Rx state, async `fetchX()` methods that call `NetworkService.get()` and parse JSON into Rx models, error handling with `ToastHelper().showErrorToast(localization.allControllerLoadError)`. Anti-pattern: `final localization = AppLocalizations.of(Get.context!)!;` is **eagerly evaluated at construction time** — if the controller is built before a context is available, this will throw. Several controllers do this.

**Architecture verdict:** Consistent, but **no repository layer** (controllers call `NetworkService` directly), **no use cases / interactors**, **no separation of data layer from presentation**. Models are JSON deserializers only (no `copyWith`, no equality, no immutability). This is fine for a small app but limits testability — there are **zero tests** in the repo (`dev_dependencies` only includes `flutter_test` and `flutter_lints`, and no `test/` directory was found).

## 7. Date/Time, Currency, Number Formatting

- **`intl: ^0.20.2`** is in `pubspec.yaml` (good).
- **`DateFormat` is used in ~25 places** — exclusively with hard-coded Gregorian patterns like `"dd MMM yyyy hh:mm a"`, `"dd/MM/yyyy"`, `"dd MMM,yyyy - hh:mm a"`, `"yyyy-MM-dd HH:mm:ss"`. **No locale-aware formatting** — none of the calls pass a `locale` argument to `DateFormat.yMMMd(localeCode)`. The `Intl.message`/`Intl.plural` APIs are not used either.
- **`NumberFormat` is NOT used anywhere** (zero matches). All money amounts are rendered via string concatenation (`"${amount} ${currencyCode}"`) with custom decimal logic from `DynamicDecimalsHelper` (returns 2 for fiat, 8 for crypto, or the site currency's `currencyDecimals` setting).
  - **No thousands separators** in the UI (amounts like `1234567.89` display as `1234567.89` instead of `1,234,567.89`).
  - **No currency symbol localization** (always shows `USD`, `EUR`, etc. — never `$`, `€`, or locale-specific symbols).
  - **No Persian-digit conversion** anywhere — amounts in fa locale will show Western digits `0-9` instead of `۰-۹`.
- **Jalali (Persian solar) calendar**: NOT supported. The date picker (`common_single_date_picker.dart` line 4) uses `flutter_rounded_date_picker` which is **Gregorian only**. There is no `shamsi_date`, `persian_datetime_picker`, `jalali_calendar` dependency. For Iranian users this is a **significant UX gap**.
- **Hijri (Islamic lunar) calendar**: NOT supported. No `hijri` package. For UAE Arabic users this is an optional-but-expected feature.
- **Date input format**: hard-coded `dd/MM/yyyy` everywhere; no respect for the user's locale (`MM/dd/yyyy` in US-style, `yyyy-MM-dd` in ISO, etc.).

**Helper files** (`lib/src/helper/`):
- `dynamic_decimals_helper.dart` — 16 lines, see above.
- `mask_email_helper.dart` — email masking.
- `toast_helper.dart` — `Fluttertoast` wrapper.

## 8. Country/Region Handling

- **`CountryController` (`lib/src/common/controller/country_controller.dart`)**: fetches from `GET /get-countries` → `CountryData { name, dialCode, code, flag, selected }`. Used at registration (personal info screen) and gated by a `country_show` register-field setting from the backend.
- **`CountryData`**: stores `dial_code` (e.g. `+98`) but **no `phone_format` / `phone_mask` / `phone_length`** field — the app cannot validate or format phone numbers per country.
- **Country selection happens at registration only** (personal info screen, line 71 of `personal_info_screen.dart`); there is no per-profile country update flow visible, and no country-based feature gating (e.g. showing Iranian payment methods to Iranian users).
- **Phone number input** (`personal_info_screen.dart` line 88): the dial code is just pre-pended to the phone field — `controller.phoneNoController.text = selectedCountry.dialCode ?? ""`. **No `libphonenumber` integration, no per-country phone validation, no formatting mask.**
- **No IP-based / SIM-based country detection** at first launch.
- **No regional routing** — all users see the same Stripe/BsiCards card providers, the same bill-payment countries, etc.

## 10. Security Audit

| # | Issue | Severity | Location |
|---|---|---|---|

## 15. Biometric / Local Auth

**`lib/src/common/services/biometric_auth_service.dart`** (52 lines):
- Uses `local_auth: ^3.0.0` (`LocalAuthentication`).
- ✅ Checks `canCheckBiometrics`, `isDeviceSupported()`, `getAvailableBiometrics()`.
- ✅ Calls `authenticate(localizedReason: ..., biometricOnly: true)`.
- The localized reason string is `localization.biometricAuthReason`.

**Biometric gate enforcement — ❌ NOT enforced on app launch.**
- `splash_controller.dart` `navigateBasedOnAuth()` only checks for connectivity and `login_current_state` — it does **not** check whether biometric is enabled and force a biometric prompt before showing the home screen.
- The biometric prompt is only triggered **manually from the sign-in screen** (`sign_in_screen.dart` line 331), and only if `isBiometricEnable.value == true`.
- **Impact**: A user who has biometric enabled, signs in, then closes the app — on relaunch they go straight to the home screen without biometric verification. Anyone holding the unlocked phone can access the wallet.

**Passcode (`/user/passcode` API, `verify_passcode` flow):**
- The passcode is a server-side 4-6 digit PIN that can be enabled/disabled via `enablePasscode`/`disablePasscode`/`verifyPasscode` endpoints.
- A `VerifyPasscodeBottomSheet` exists (`lib/src/presentation/widgets/verify_passcode_bottom_sheet.dart`) but I see no evidence it's gated on app launch either.
- `verify_passcode_controller.dart` exists in `lib/src/common/controller/`.

**Recommendations:**
1. Add a biometric / passcode gate in `SplashController.navigateBasedOnAuth()` — if `isBiometricEnable.value == true` and `login_current_state == "logged_in"`, route to a `biometric_gate` screen that calls `BiometricAuthService.authenticateWithBiometrics()` and only proceeds on success.
2. Add a `session_timeout` so the gate re-triggers after N minutes of inactivity.
3. **Stop storing the plaintext password** — biometric re-login should use a refresh token (server-issued, short-lived) or a server-side biometric challenge, not a stored password.
